## 一、先看一手证据：你的消融实验已经把问题说透了

这不是我推测的，是你自己跑出来的数据（`experiments/results/ablation/*_summary.json`，n=100）：

### GSM8K（算术题）

| 配置 | EM | 相对 Full |
|------|-----|----------|
| Full GERS | 0.61 | 基准 |
| w/o Decompose（退化为 CoT） | **0.74** | **+13pt** |
| w/o Constraint（去掉约束解码） | **0.80** | **+19pt** |
| w/o Context | 0.64 | +3pt |
| w/o Feedback | 0.61 | 0pt |

### HotpotQA（多跳问答）

| 配置 | EM | 相对 Full |
|------|-----|----------|
| Full GERS | 0.43 | 基准 |
| w/o Decompose（退化为 CoT） | 0.35 | -8pt |
| w/o Context | 0.40 | -3pt |
| w/o Constraint | 0.43 | **0pt** |
| w/o Feedback | 0.43 | **0pt** |

### 从这两张表能直接读出的硬事实

1. **「分解 + DAG」只在多跳任务上有效**：HotpotQA 去掉分解掉 8pt（有用），但 GSM8K 去掉分解反而涨 13pt（有害）。说明 GERS 对「单步算术题」是过度拆解、引入噪声。
2. **约束解码（ConstrainedDecoder）是废件甚至负资产**：HotpotQA 上去掉它 EM 一点不变（0.43 == 0.43），GSM8K 上去掉它反而暴涨到 0.80。**这个模块在两个数据集上都没有正贡献。**
3. **闭环修正（FeedbackLoop）完全没触发**：w/o Feedback == Full（两个数据集都是），因为代码里 `max_iterations=1`，修正分支根本进不去。**这是一个写了但从未运行的模块。**
4. **依赖传递（Context）是有效的**：去掉它两个数据集都掉分（GSM8K -3、HotpotQA -3），这是真正起作用的设计之一。

> 结论：GERS 名义上有「分解 / 拓扑执行 / 依赖传递 / 约束解码 / 一致性校验 / 闭环修正」六件套，**实际有效的只有「分解（仅多跳）+ 拓扑执行 + 依赖传递」三件**，其余三件是装饰或负担。

## 二、阻断级问题（不修就不能发，按严重度排序）

### 🔴 P0-1：EM 评估指标有 bug，GSM8K 全部数据不可信

`src/utils/metrics.py` 的 `exact_match`：

```python
# 部分匹配：预测是参考的子串，或参考是预测的子串
if pred and ref and (pred in ref or ref in pred):
    return 1.0
```

这个「双向子串包含」对数值答案是灾难：
- 参考答案 `18`，模型输出 `180` → `"18" in "180"` = True → **错判为对**
- 哪个方法输出越啰嗦、越容易包含到正确数字片段，EM 就越虚高

**影响**：ToT 修复后 GSM8K 飙到 0.86（比 CoT-SC 的 0.762 还高一大截）这个反常结果，极可能就是子串匹配喂出来的假高分。**当前 GSM8K 的全部横向对比都不可信。**

**修法**：
- GSM8K → 抽取答案中最后一个数字，做**数值相等**比较（标准 `####` 格式 / 末位数字）
- HotpotQA / CLUTRR → 用 SQuAD 式 normalized EM（去冠词、去标点、小写），**删掉双向子串那一行**
- 修完用现有结果 JSON 里保存的 `raw_response` **离线重新算分**，不用重跑模型，几分钟出真数据

### 🔴 P0-2：三份文档三套数据，自相矛盾

- `docs/experiment_records.md`（6/24）：ToT = 0.20 / 0.02，结论「GERS 全面最优」
- `experiments/results/*.json`（最新）：ToT = 0.86 / 0.43
- `docs/paper_draft.md`：改成了「效率换性能」叙事

三套口径并存，自己都会搞混，审稿人一眼看穿。**必须在修完 P0-1 后用同一套指标统一刷新这三处。**

### 🔴 P0-3：样本量不一致，主表不可比

主实验里 GERS / CoT / CoT-SC 是 n=500，ToT 是 n=100。拿 100 条的 0.86 和 500 条的方法并列进同一张表，统计上不成立。**要么 ToT 补到 500，要么主表统一到同一 n。**

## 三、方法层面的实现问题（GERS 本身怎么改）

### 问题 1：约束解码（ConstrainedDecoder）应当直接砍掉或重做

**现状**：消融证明它零贡献/负贡献。看实现就知道为什么——它只是把图的路径描述拼成一段文本塞进 prompt（`prompt_builder.py` 的 `[Reasoning Path]`），既没有真正约束解码空间（不是 logits 级约束），又增加了 prompt 长度和噪声。它叫「ConstrainedDecoder」但其实啥也没约束。

**改法（二选一）**：
- **A（推荐，省事）**：直接删掉这个模块，论文里不再宣称「约束解码」。少一个被消融打脸的组件，比硬留着强。
- **B（想保留卖点）**：重做成**真正的结构约束**——比如强制子答案必须引用前驱节点的答案（输出格式校验 + 不满足就重生成），让它从「装饰文字」变成「可验证的硬约束」。

### 问题 2：闭环修正（FeedbackLoop）要么真正激活，要么删掉

**现状**：`max_iterations=1` 写死，`if score < threshold and self.max_iterations > 1` 永远进不去。这是一个存在于代码里但从未执行的死分支。

**改法**：
- 如果想保留：默认 `max_iterations=2~3`，把 `consistency_threshold` 调低到真正能触发的水平，然后**专门做一组「修正前 vs 修正后」对照实验**，证明它能涨分。涨不动就老实删掉。
- 关键设计：修正时不要重跑整条链，而是**只重做 Consistency Score 最低的那个子问题节点**（局部修正），这样成本可控、效果可归因。

### 问题 3：一致性校验（Consistency Score）算了但没用上

**现状**：`enable_nli=False` 默认关闭，分数恒在 0.6~0.65 附近**没有区分度**；而且算完直接塞进返回值，不影响任何输出决策。一个号称「图约束 + 一致性校验」的方法，校验结果对答案零影响——这是方法叙事的核心断裂。

**改法（这是 GERS 最有潜力的升级方向）**：
把 Consistency Score 从「事后报告」变成「生成时的选择信号」：

> **图级别的 Self-Consistency（强烈推荐做这个）**
> - 对同一问题生成 K 条不同的推理 DAG（不同 temperature / 不同分解）
> - 用 Consistency Score 给每条图打分，**选得分最高的那条**作为最终答案
> - 这直接对标 CoT-SC：CoT-SC 用「答案多数投票」，你用「图结构质量打分」
> - 卖点一句话讲清：**结构化质量信号 > 朴素多数投票**

这个改动的好处：
1. 让 Consistency Score 第一次产生真实价值（不再是装饰）
2. 直接对标并有机会打败最强基线 CoT-SC
3. 论文叙事从「我有一致性校验」升级成「我用图结构做了更聪明的 self-consistency」

为了让它有效，`enable_nli` 要真正打开，让分数有区分度——否则全是 0.6 没法选。

### 问题 4：自适应复杂度判断要变成「显式策略」而非「隐藏开关」

**现状**：`adaptive=True` 时简单题跳过分解。这个设计本身是对的（GSM8K 消融已经证明「不分解」在算术题上更好），但它现在藏在代码里，没有作为方法贡献被讲出来。

**改法**：把它**前置成论文里的一个正式贡献点**——「复杂度自适应路由：简单题走直接 CoT，复杂题走图约束推理」。配一组「always-decompose vs adaptive」的对照，用 GSM8K 那个 13pt 的差距说明路由的价值。这是你现成的、有数据支撑的卖点，别浪费。

## 四、GERS 真正的「立足点」在哪里（最重要的战略判断）

把分题型数据摆出来（`hotpotqa_type_analysis.json`，n=500）：

| 方法 | bridge 题 EM | comparison 题 EM |
|------|-------------|-----------------|
| Zero-Shot | 0.327 | 0.479 |
| Standard CoT | 0.270 | 0.500 |
| **GERS** | 0.337 | **0.677** |

**这张表是你整个项目最值钱的证据**：
- bridge（链式多跳）题：GERS 0.337 vs CoT 0.270，只领先一点点
- **comparison（对比型）题：GERS 0.677 vs CoT 0.500，领先 17.7pt** ← 这才是 GERS 的杀手锏

**原因很清楚**：对比型问题（"A 和 B 谁更…"）天然是「两条独立子链 + 一个汇合节点」的 DAG 结构，GERS 能把对比双方拆成并行分支分别求解再汇总，而线性 CoT 和盲目树搜索（ToT）做不到这种结构化分解。

**战略建议**：
1. **把论文主线从「通用多跳推理」收缩到「依赖结构密集 / 对比型推理」**。不要再跟 ToT 比谁 GSM8K 高——那是 GERS 的劣势战场。
2. **换数据集放大优势**：用 **2WikiMultiHopQA**、**MuSiQue** 这类依赖更深、对比/桥接结构更复杂的数据集。在这些任务上 ToT 的盲目搜索会爆炸，GERS 的拓扑排序优势才显出来。
3. **砍掉自造 CLUTRR**：程序化生成的 300 条，所有方法 EM 都在 0.13~0.19 贴地，零区分度，留着只暴露弱点。

## 六、一段话给你定个调子

你这套 GERS 不是没价值，问题是**你把六个组件都当卖点在守，结果一半被自己的消融实验打脸**。真正聪明的做法是「断舍离」：

- **砍掉**约束解码、（当前形态的）闭环修正——它们是负债
- **保留并强化**分解 + 拓扑执行 + 依赖传递——这是真有用的核心
- **升级**一致性校验为「图级 Self-Consistency 选择」——这是从「能用」到「能打」的关键一跃
- **聚焦**对比型/依赖密集推理——这是 GERS 唯一能稳赢 CoT/ToT 的战场

修指标 → 砍废件 → 把 Consistency Score 用起来 → 收缩战场到对比型多跳。按这个顺序走，这篇东西能从「自相矛盾」变成「小而锋利」。
