# GERS 方法创新改进方案：子答案双向交叉验证

> 基于HotpotQA 500条真实诊断 + 代码精读，直击 Consistency Score 无区分度这一核心瓶颈。
> 设计原则：保留现有 GERS/GERS-SC 完整逻辑不变，新增"反向验证"模块 + 改造 CS 加权，实现方法层创新。

---

## 一、问题诊断（500 条已验证）

| 诊断项 | 数据 | 结论 |
|--------|------|------|
| GERS vs StdCoT 对错分布 | GERS对CoT错=40 / CoT对GERS错=31 / 双错=328 / 双对=101 | 净+9，500条不显著（p=0.275） |
| CS 全局分布 | mean=0.662, std=0.098；0.6分312条、0.7分150条 | 高度聚集，无分层 |
| CS 对错区分度 | 答对 CS=0.6592, 答错 CS=0.6628, 差=**−0.0035** | 反向、无区分，等价随机 |
| GERS-SC vs GERS+自适应 | 0.282 vs 0.284, p=1.000 | 图级自一致性无增益 |

**根因**：现有 `Consistency Score = 0.4·连通性 + 0.3·无环 + 0.3·覆盖度` 是**纯图论结构指标**，只校验 DAG 是否合法（连通、无环、覆盖），不校验推理内容。而 LLM 生成的分解图几乎都是合法 DAG，故所有样本得分趋同（0.6~0.7），Graph-SC 的"多采样选 CS 最高"退化为随机抽取。

**唯一经检验的 GERS 优势**：comparison 子集 +10.5pt（McNemar p=0.013），但被 404 条持平的 bridge 题稀释。

## 二、方向 1：子答案双向交叉验证（主推，核心创新）

### 2.1 核心思想与创新点

**现有正向流程**：问题 → DAG 分解 → 拓扑序执行 → 子答案 {a_i} → 汇总 → 最终答案 A。

**新增反向验证流程**：以"最终答案 A + 原始上下文 C"为锚，**反向逐个重答每个子问题**，得反向子答案 {a'_i}；对比正向 {a_i} 与反向 {a'_i}，不一致则该子问题可信度降权。

**创新性（DAG 独有，线性 CoT/ToT 无法实现）**：
- 线性 CoT 是单链，没有可独立校验的子问题结构，无法做"正反向双向校验"。
- GERS 的 DAG 把推理拆成有依赖的子问题节点，天然支持"正向执行 + 反向验证"的双向一致性校验。
- 这让 Consistency Score 从"图是否合法"升级为"推理内容是否自洽"，**真正反映推理质量**。

文献支撑：EMNLP 2024《LLMs Can Self-Correct with Key Condition Verification》的关键条件反向自校正思路。

### 2.2 新 Consistency Score 公式

$$CS_{new} = 0.3 \cdot S_{struct} + 0.7 \cdot S_{crossval}$$

- $S_{struct}$：现有结构分（连通性+无环+覆盖度），保留作为图合法性基础分（0~1）。权重降至 0.3，因其无区分度。
- $S_{crossval}$：**正反向子答案一致性分**（0~1），新增，作为内容自洽性主信号，权重 0.7。

$$S_{crossval} = \frac{1}{n}\sum_{i=1}^{n} w_i \cdot \mathbb{1}[\text{match}(a_i, a'_i)]$$

其中 $w_i$ 为子问题权重（下游/汇合节点权重更高，因其错误传播影响更大），$\text{match}$ 为语义匹配判断。

### 2.3 核心代码改造逻辑

#### 改造点 1：新增 `_backward_verify()` 方法（`generation_pipeline.py`）

```python
def _backward_verify(self, question: str, context: str,
                     final_answer: str, sub_qa_chain: list) -> Dict:
    """
    反向验证：以最终答案A + 上下文C为锚，反向重答每个子问题，
    对比正向子答案 a_i 与反向子答案 a'_i 的一致性。

    Returns:
        {
            "crossval_score": float,         # 正反向一致性分 [0,1]
            "verifications": [{sub_question, forward_answer, backward_answer, match}],
        }
    """
    if not sub_qa_chain or not final_answer:
        return {"crossval_score": 0.5, "verifications": []}  # 无子问题退化中性分

    context_section = f"\nContext: {context[:1500]}" if context else ""
    verifications = []
    match_count = 0
    total_weight = 0.0

    for i, item in enumerate(sub_qa_chain):
        sub_q = item["sub_question"]
        forward_ans = item["sub_answer"]
        # 权重：下游子问题（i越大）权重越高，错误传播影响更大
        weight = 0.5 + 0.5 * (i + 1) / len(sub_qa_chain)

        prompt = BACKWARD_VERIFY_PROMPT.format(
            original_question=question,
            context_section=context_section,
            final_answer=final_answer,
            step_index=i + 1,
            sub_question=sub_q,
        )
        try:
            resp = self.model.generate(prompt, max_tokens=150, temperature=0.0)
            backward_ans = self._extract_sub_answer(resp)  # 复用现有提取
        except Exception:
            backward_ans = ""

        matched = self._answers_match(forward_ans, backward_ans)
        if matched:
            match_count += weight
        total_weight += weight
        verifications.append({
            "sub_question": sub_q,
            "forward_answer": forward_ans,
            "backward_answer": backward_ans,
            "match": matched,
        })

    crossval_score = match_count / total_weight if total_weight > 0 else 0.5
    return {"crossval_score": round(crossval_score, 4), "verifications": verifications}
```

#### 改造点 2：新增反向验证 Prompt

```python
BACKWARD_VERIFY_PROMPT = """You are verifying a reasoning step in reverse.

Original question: {original_question}
{context_section}
The final answer to the original question is known to be: {final_answer}

Now, given this final answer and the context, re-derive the answer to this
specific sub-question independently:

Sub-question (Step {step_index}): {sub_question}

Answer this sub-question based ONLY on the context (do not just copy the final
answer). If the final answer is correct, your sub-answer should be consistent
with it; if the final answer is wrong, your independent sub-answer may differ.
End with:
Sub-answer: <your answer>"""
```

**关键设计**：prompt 要求"基于上下文独立重答，不要照抄最终答案"——缓解 confirmation bias（模型迎合 A 的倾向）。这让反向验证在 A 错时仍能给出独立判断，捕获推理链与上下文的不自洽。

#### 改造点 3：一致性判断 `_answers_match()`

```python
def _answers_match(self, a: str, b: str) -> bool:
    """判断正反向子答案是否语义一致。先字符串匹配，不匹配再用 LLM 判断。"""
    if not a or not b:
        return False
    na = normalize_for_vote(a)  # 复用现有归一化
    nb = normalize_for_vote(b)
    if na == nb:
        return True
    # 包含关系（短答案是长答案的子串）
    if na and nb and (na in nb or nb in na):
        return True
    # 数值相等
    import re
    nums_a = re.findall(r'-?\d+\.?\d*', a)
    nums_b = re.findall(r'-?\d+\.?\d*', b)
    if nums_a and nums_b and abs(float(nums_a[-1]) - float(nums_b[-1])) < 1e-6:
        return True
    # 仍不匹配：LLM 语义判断（可选，成本敏感时可关闭）
    if self.enable_llm_match:
        return self._llm_answers_match(a, b)
    return False
```

**分级策略**：默认 `enable_llm_match=False`（成本敏感），靠字符串/数值匹配；若区分度仍不足，再开启 LLM 判断（每不匹配对 +1 次 LLM）。

#### 改造点 4：集成到 `reason()` 主流程（⑥ 之后）

在 `reason()` 现有 ⑥ 一致性校验之后，插入反向验证并重算 CS：

```python
# ── ⑥ 一致性校验（现有结构层）──
check_result = self.consistency_checker.check(graph, reasoning_text)
struct_score = check_result["structural_score"]
score = check_result["consistency_score"]

# ── ⑥.5 反向验证（新增内容层）──
if self.enable_backward_verify and sub_qa_chain:
    cv = self._backward_verify(question, context, answer, sub_qa_chain)
    crossval_score = cv["crossval_score"]
    # 新 CS = 0.3·结构 + 0.7·正反向一致性
    score = 0.3 * struct_score + 0.7 * crossval_score
    check_result["crossval_score"] = crossval_score
    check_result["crossval_detail"] = cv["verifications"]
    check_result["consistency_score"] = round(max(0.0, min(1.0, score)), 4)
```

#### 改造点 5：Graph-SC 自动受益（无需改 `_reason_with_self_consistency`）

`_reason_with_self_consistency`（line 440）的 `best = max(candidates, key=lambda c: c["consistency_score"])` **代码不变**。因为每条候选 DAG 的 CS 现在由 `reason()` 内的新公式计算（含 crossval），CS 有了区分度后，选优自动生效——这正是修复的核心。

### 2.4 与现有代码的兼容性

| 现有逻辑 | 是否改动 |
|---------|---------|
| `reason()` ①-⑤ 分解/构图/执行/汇总 | **不变** |
| `_reason_with_self_consistency` SC 选优 | **不变**（自动受益） |
| `consistency_score.py` 结构校验 | **不变**（仍算 $S_{struct}$） |
| `_extract_sub_answer` | 复用（反向验证提取） |
| `normalize_for_vote` | 复用（一致性判断） |
| 新增 | `_backward_verify`, `_answers_match`, `BACKWARD_VERIFY_PROMPT`, `reason()` 内 ⑥.5 插入, `__init__` 加 `enable_backward_verify` 参数 |

**改动量小**：1 个新方法 + 1 个新 prompt + 1 个匹配函数 + `reason()` 内 ~6 行插入 + 1 个构造参数。现有框架完全兼容。

### 2.5 成本分析

| 方法 | 现有 LLM 调用/题 | 反向验证后增加 | 总计 |
|------|:-:|:-:|:-:|
| GERS+自适应 | 1~5 | +n（子问题数，均~3） | 4~8 |
| GERS-SC (K=3) | 12~18 | +3n（均~9） | 21~27 |

成本增加但可控。反向验证的 n 次子问题调用**可并行化**（无依赖），用线程池并行执行，wall-clock 仅增 ~1 次调用时间。`enable_backward_verify` 作为开关，可单独消融。

### 2.6 预期效果与风险

**预期**：
- $S_{crossval}$ 对错误推理链会低（正反向不一致），对正确推理链高 → CS 区分度从 −0.0035 提升到 **+0.05~+0.15（正向）**。
- Graph-SC 选优生效，在 comparison + 部分 bridge 题上提升。
- 预期 GERS-SC 整体 F1 提升 2~5pt，可能达到显著。

**风险与缓解**：
1. **Confirmation bias**（反向迎合错误 A）→ prompt 强制"基于上下文独立重答，勿照抄 A"；必要时增加"不告知 A 的独立重答"作为第二锚点。
2. **反向验证本身出错**→ 仅作 CS 信号（降权），不直接修改答案，错误影响有限。
3. **成本翻倍**→ 子问题并行 + `enable_llm_match` 分级开关。
4. **A 错时整体自洽假象**→ 这是反向验证的固有局限，需配合方向 2（置信度）缓解；但即便如此，对"推理链内部不自洽"的捕获已远胜纯结构 CS。

## 三、方向 2：子答案置信度加权汇总（次优先，低成本补充）

**现状**：`generation_pipeline.py` 已预留 `_estimate_answer_confidence()` 等置信度逻辑但未完整启用。所有子答案平等加权汇总。

**改造**：在 ⑤ 汇总阶段，基于子步骤置信度动态加权。低置信子答案在汇总 prompt 中标注"uncertain"，降低其对最终答案的影响，减少错误传播。

```python
# ⑤ 汇总时，对低置信子答案标注
for item in sub_qa_chain:
    conf = self._estimate_answer_confidence(item["sub_answer"], item["raw_response"])
    if conf < threshold:
        reasoning_chain += f"\n[Note: Step answer '{item['sub_answer']}' has LOW confidence]"
```

**优势**：改动极小，可快速验证。与方向 1 正交（方向 1 改 CS 选择，方向 2 改汇总生成）。

## 四、方向 3：动态自适应分解深度（第三优先级）

**现状**：`_judge_complexity()` 仅二分 simple/complex，复杂题统一固定跳数分解。

**改造**：前置判断问题所需推理跳数（2 跳 / 3-4 跳），在 `_decompose()` prompt 中指定目标子问题数，动态控制分解粒度。避免过度分解（放大错误）或分解不足（缺失步骤）。

**风险**：跳数判断本身可能错，需小样本验证。优先级最低，在前两方向验证后再考虑。

## 五、实施路线图

### 阶段 1：方向 1 落地 + 小样本验证（核心）
1. 实现 `_backward_verify` + `_answers_match` + `BACKWARD_VERIFY_PROMPT`
2. `reason()` 内插入 ⑥.5，`__init__` 加 `enable_backward_verify=True`
3. **小样本验证（50 条 HotpotQA）**：跑 gers_sc，检查
   - $S_{crossval}$ 对错区分度是否转正（目标 > +0.05）
   - CS 整体分布是否拉开（不再聚集 0.6/0.7）
   - GERS-SC vs GERS+自适应 是否出现差距
4. **若区分度转正** → 进入阶段 2；**若仍无区分** → 诊断反向验证 prompt/匹配逻辑

### 阶段 2：500 条全量验证
5. 跑 gers_sc + gers_adaptive 500 条（`enable_backward_verify=True`）
6. 算显著性：GERS-SC vs CoT-SC 的 bootstrap CI + McNemar

8. 启用置信度加权汇总，重跑验证增量

### 阶段 4：论文更新
9. 主表用 500 条新结果，主打"双向交叉验证修复 CS 区分度"创新点
10. 消融：w/o 反向验证（退回旧 CS）vs w/ 反向验证，证明 crossval 贡献

## 六、为什么方向 1 是最优选择

1. **直击根因**：CS 无区分度是 GERS-SC 失效的唯一根因，方向 1 从"内容校验"取代"结构校验"，根本解决。
2. **创新性强**：DAG 双向交叉验证是图结构推理独有，线性 CoT/ToT 无法实现，形成清晰差异化卖点。
3. **改动量小**：新增 1 方法 + 1 prompt，现有逻辑全保留，风险可控。
4. **可消融**：`enable_backward_verify` 开关，便于论文证明贡献。
5. **可验证**：CS 区分度是量化指标，小样本 50 条即可判断方向是否成立，不必盲跑 500 条。
